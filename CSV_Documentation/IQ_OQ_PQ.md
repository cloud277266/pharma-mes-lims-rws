## 1. 운전 적격성 평가 (OQ: Operational Qualification)

### [OQ-LIMS-001] Audit Trail 변경 사유 강제화 테스트
*   **관련 요구사항**: REQ-LIMS-001
*   **테스트 목적**: 품질 데이터 수정 시 `changeReason` 필드를 고의로 누락했을 때, 시스템이 데이터 수정을 차단하고 적절한 오류를 반환하는지 검증한다.
*   **테스트 절차**:
    1. LIMS 시스템의 특정 실험 결과를 수정하는 API(PUT `/api/results/:id`)를 호출한다.
    2. Request Body에 수정할 데이터는 포함하되, `changeReason` 파라미터는 포함하지 않는다.
*   **기대 결과**: 서버는 데이터 수정을 거부하고, HTTP 400 Bad Request 상태 코드와 함께 "변경 사유(changeReason)가 누락되었습니다"라는 에러 메시지를 반환해야 한다.

#### 테스트 결과: PASS
*   **테스트 일시**: 2026-05-17
*   **테스트 환경**: Mini LIMS v0.1.0 / Next.js 16 / MySQL 8.0 (Docker)
*   **실행 명령**:
    ```bash
    # changeReason 누락하여 PUT 요청
    curl -X PUT http://localhost:3000/api/results/4 \
      -H "Content-Type: application/json" \
      -d '{"value": 99.5}'
    ```
*   **실제 응답**:
    ```json
    HTTP/1.1 400 Bad Request
    {"error":"변경 사유(changeReason)가 누락되었습니다"}
    ```
*   **판정**: 기대 결과와 실제 결과 일치 — **PASS**
*   **증적 자료**: 감사 추적(Audit Trail) 화면에서 변경 사유 없이 수정 불가 확인 (브라우저 수정 모달의 변경 사유 필드 `required` 속성 적용)

### [OQ-LIMS-002] Maker-Checker 워크플로우 테스트
*   **관련 요구사항**: REQ-LIMS-002
*   **테스트 목적**: 데이터 입력자(Maker)가 본인이 입력한 데이터를 스스로 승인하려 할 때 시스템이 이를 차단하는지 검증한다.
*   **테스트 절차**:
    1. `maker01` 계정으로 로그인하여 시험 결과(RAW-2026-001, pH 측정)를 입력한다.
    2. `checker01` 계정(CHECKER 역할)으로 재로그인한다.
    3. 타인이 입력한 데이터에 대해 승인(APPROVE) 요청을 수행한다.
    4. 동일 계정이 본인 데이터를 승인하려는 시나리오를 API로 직접 시도한다.
*   **기대 결과**:
    - 정상 승인(타인 데이터): HTTP 200, 상태가 APPROVED로 변경
    - 본인 데이터 자가 승인 시도: HTTP 403, "본인이 입력한 데이터는 승인할 수 없습니다" 반환

#### 테스트 결과: PASS
*   **테스트 일시**: 2026-05-17
*   **테스트 환경**: Mini LIMS v0.1.0 / Next.js 16 / MySQL 8.0 (Docker)
*   **검증 결과 요약**:

    | 시나리오 | 수행자 | 대상 데이터 | HTTP 응답 | 판정 |
    |---|---|---|---|---|
    | 정상 승인 | checker01 (CHECKER) | RAW-2026-001 pH 측정 (maker01 입력) | 200 OK / status: APPROVED | PASS |
    | 정상 승인 | checker01 (CHECKER) | RAW-2026-001 수분함량 (maker01 입력) | 200 OK / status: APPROVED | PASS |
    | 정상 반려 | checker01 (CHECKER) | FIN-2026-001 붕해시간 (maker01 입력) | 200 OK / status: REJECTED | PASS |
    | 자가 승인 차단 | maker01 (MAKER) | 본인 입력 데이터 승인 시도 | 403 Forbidden | PASS |
    | 권한 없는 승인 차단 | maker01 (MAKER) | CHECKER 전용 기능 접근 | 403 Forbidden | PASS |

*   **Audit Trail 기록 확인**: 승인/반려 모든 작업이 감사 추적에 APPROVE/REJECT 액션으로 자동 기록됨
*   **판정**: 기대 결과와 실제 결과 일치 — **PASS**

### [OQ-RWS-001] 칭량 허용 오차 차단 테스트
*   **관련 요구사항**: REQ-RWS-001
*   **테스트 목적**: 실제 칭량값이 허용 오차 범위를 벗어난 상태에서 작업을 강제 완료하려고 할 때 시스템이 이를 차단하는지 검증한다.
*   **테스트 절차**:
    1. RWS Simulator에서 바코드 `BC-RAW-002` 스캔 (유당, 목표 100g ±1%)
    2. 실측값 `98` 입력 → 완료 버튼 상태 확인
    3. 실측값 `102` 입력 → 완료 버튼 상태 확인
    4. 실측값 `100.5` 입력 → 완료 버튼 상태 확인 후 완료 진행
*   **기대 결과**: 오차 범위 초과/미달 시 '칭량 완료' 버튼이 비활성화되고 "허용 오차를 벗어나 완료할 수 없습니다 (REQ-RWS-001)" 경고 문구가 표시되어야 한다.

#### 테스트 결과: PASS
*   **테스트 일시**: 2026-05-17
*   **테스트 환경**: RWS Simulator v0.1.0 / C# WPF / .NET 10 / Windows 10
*   **검증 결과 요약**:

    | 바코드 | 목표(g) | 허용범위 | 실측값(g) | 판정 표시 | 완료 버튼 | 결과 |
    |---|---|---|---|---|---|---|
    | BC-RAW-002 | 100.0 | 99.00 ~ 101.00 | 98.0 | 미달 (적색) | 비활성화 | PASS |
    | BC-RAW-002 | 100.0 | 99.00 ~ 101.00 | 102.0 | 초과 (적색) | 비활성화 | PASS |
    | BC-RAW-002 | 100.0 | 99.00 ~ 101.00 | 100.5 | 적합 (녹색) | 활성화 | PASS |
    | FAKE-001 | - | - | - | 바코드 불일치 차단 | 비활성화 | PASS |

*   **Hard-lock 동작 확인**: `CanComplete` 프로퍼티가 `IsBarcodeVerified AND IsWithinTolerance` 조건을 만족할 때만 `true` → WPF Command `CanExecute`에 바인딩되어 버튼 자동 비활성화
*   **라벨 발행 확인**: 적합 범위 내 완료 시 칭량 완료 라벨(작업 지시 번호, 원료 바코드, 원료명, 목표/실측 중량, 작업자, 발행 일시) 팝업 정상 출력
*   **판정**: 기대 결과와 실제 결과 일치 — **PASS**
